Lab 1 TFTP - Socket programming

Motivation

After completing this lab you should be able to implement and make design choices regarding the basics of socket programming in the form of a functioning TFTP client. It will also demonstrate the basics of performance measurements.

This lab addresses the learning outcome "Design and implement communicating applications". The purpose of this assignment is that you get hands on experience and practice network programming, specifically socket programming.

Instructions

This assignment needs to be performed in groups of two. You are encouraged to use the provided resources and to discuss with other groups. However, you are not allowed to use code from other groups doing the assignment. If you discuss with another group, please state so in the upload area (as a comment) to give them credit and avoid suspicion of cheating.

There are two lab sessions (non-mandatory) dedicated to this lab, we highly recommend that you come prepared, make use of them as much as possible and ask questions. Expect that you will need to work on this assignment outside of the scheduled lab hours. At the third session, you will have to present your results.

The assignment is made in an open way with no step-by-step instructions. You should be able to take in all the necessary information to plan and design your work and solution. Below follows information that is essential in completing this lab.

This lab involves writing a TFTP client in C or Python and doing some simple measurements. TFTP stands for Trivial File Transfer Protocol and is the predecessor to FTP. It is still used today for transferring new firmware to routers and other embedded devices and also for booting network terminals. It is a simple protocol and therefore straightforward to understand. It has two main uses:

  1.  Transferring a file from a client to server.
  2.  Transferring a file from a server to a client.

Your task will be to write code for a client (you do not have to implement a server) that can perform both of these functions. In other words, you need to write code that can PUT a file from your client onto a server and GET a file from a server to your client.

The first step in this excercise is to read RFC1350 which is the document that details the TFTP standard. The RFC document contains all the information you need about how TFTP works, so it is essential that you read and understand it before starting to program. You may also want to read up on socket programming, e.g., using this intro.

Detailed Instructions

You can select between doing this exercise in C or Python. In either case, you need to base your software on the socket interface. Start by downloading the appropriate code skeleton and language-dependent instruction files:

  • C-language files (tftp_c.tar.gz)
  • Python files (tftp_python.tar.gz)

You find them in the file area Lab 1 Files.

Then make your software code.

Testing your Software

One of the major tasks in this assignment is actually testing and debugging your code. You are encouraged to also work outside scheduled time. However, we recommend you to do your testing on the schools UNIX system since it is there you have to demonstrate your code. You can login remotely using ssh (ask a TA or a friend if you do not know how).

We strongly advise against testing your code when connected to a wireless network since packet loss can be severe, making it difficult to debug.

For testing, there is a TFTP server running on the machine rabbit.it.uu.se which is accessible both inside and outside the university network. You can access it using different port numbers to test different behavior. The following ports are open:

  • 69 - The standard port, reserved for TFTP. Only accessible inside the university network.
  • 6969 - Same as 69 but accessible outside the university network as well.
  • 10069 - Generates packet loss, you need to send a large enough file to get some packet loss. (Accessible both inside and outside the university network)
  • 20069 - Generates duplicates, you need to send a large enough file to get duplicates. (Accessible both inside and outside the university network)

If you wish to retrieve a file, there are three test files available on the server: small.txt, medium.pdf, large.jpeg. If you wish to upload something feel free. However, do not upload objectionable material as all TFTP traffic is logged and offenders will be reported.

The first thing to check when completing a transfer is that the file size of the original and the transferred file are exactly the same. Then verify that the md5 of your transferred file is the same as the original. If two files have the same md5 they are exactly the same. The Unix command is md5sum. Both the md5 and file sizes of the files are listed in the table below.

Wireshark network protocol analyzer (or other such programs) is a very good way to debug your code and especially that you format the packets correctly. However, due to privacy and user policy issues it is not available on the school Linux systems, so if you want to use it you need to download it to your own laptop. Run Wireshark and capture traffic while running your tftp client. Select a captured UDP packet and right-click and choose "Decode as" then select TFTP. That way you can see how every packet you send out "actually" looks on the network as well as read the TFTP related info. If Wireshark reports "malformed packet" for the packets that you have sent out, then you are doing something wrong. Often it is a case of a trailing zero. Remember, try and avoid a wireless connection when doing this.

A text-based alternative to Wireshark is tcpdump.

Do Some Performance Measurements

Now when your client is ready, you need to perform some small measurements. The target should be to analyze the time it takes from the download to completion (e.g., measure the response time) for various scenarios. To measure the time, you can use the built in command time as follows:

all three files and the three ports (69, 10069, 20069) and analyze the differences.

   bash$ time ./tftp-mysolution -g small.txt joshua.it.uu.se

   real    0m0.324s
   user    0m0.004s
   sys    0m0.044s

Collect the real time from about 10 repetitions (or more if needed). Calculate the mean and the confidence intervals. See if you can draw any conclusions from these measurements.

Exactly which scenarios you want to measure is up to you, but you can experiment with the three filesizes, upload/download, differences between the amount of packet loss (by choosing the different ports). Running the clients from the university computers, from home, or over wireless. Etc. You need to run a couple of scenarios and make some comparisons between them.

You should produce a table or graph with the result and one paragraph of text explaining it and arguing for your conclusions. This report should be shown at the demonstration.

Note that your first run will take longer time to complete. This is likely to be due to DNS lookups and loading of the code into memory, etc. Discard such trails.

Presentation of Solution

You should upload all the required software files to run your TFTP client in the Upload Lab 1 Files assigment area found here in the left column just below this entry. Make sure you first go through the check list before uploading to make sure you have solved everything.

You, together with your lab partner, will have to orally present your solution to a teacher (assistant), demonstrate its working, and show the results of your measurements campaign during about 15 minutes. This will be done in a UNIX lab room. You should be prepared to explain all your code.

Make sure both of you have access to the code in case your lab partner does not show up. Also make sure you prepare the demonstration before your time slot begins.

Passing Criteria

To pass this lab, your program must compile without errors and warnings, not crash under any circumstances, handle all error situations, and otherwise follow the TFTP protocol standard. It must implement both PUT and GET of files and also deal with lost and duplicate packets (i.e., port 10069 and 20069 on the server).

You (both of you) must be able to explain how the code works. The code must be based on the provided code-skeleton. You should be able to show some measurement based on your code from several scenarios and draw some conclusions from this.

Deadline

You need to hand in the finished source code on Wednesday 11 November 23:59.

Presentation of the program and measurements will take place on Monday 16 November 8:15-10:00 in P2507. Time schedule for when you should demonstrate your solution will appear later.
